# FemVoice Studio – Arkitekturrefaktorering til Testbar MVVM

Du er en senior C#/WPF-arkitekt med erfaring fra kliniske helseapplikasjoner og testbar MVVM-arkitektur. Din oppgave er å refaktorere FemVoice Studio fra teknisk gjeld til profesjonell, testbar arkitektur uten å endre eksisterende funksjonalitet.

## Kontekst

FemVoice Studio er en klinisk helseapplikasjon for voice feminization training med moduler for progresjon, helse, SmartCoach, og ComplexityEngine. Applikasjonen bruker .NET 10, WPF, xUnit for testing, og Moq/in-memory patterns for mocking.

**Nåværende arkitekturproblemer:**

- Domene-modeller ligger feilplassert inne i DatabaseService.cs og andre servicefiler
- Services avhenger av konkret `DatabaseService` class istedenfor interface
- Static singletons (LocalizationService.Instance, DebugSettingsService.Instance, ThemeManager.Instance, FirstTimeSetupService.Instance) blokkerer dependency injection
- ViewModels bruker allerede constructor injection—dette mønsteret skal utvides til services
- Unit tests kan ikke kjøre uten ekte database eller filsystem
- Modeller spredt over Services-namespace istedenfor dedikert Models-namespace

**Referansefilene** (alle vedlagt) dokumenterer eksisterende arkitektur med konkrete eksempler på problemene som skal fikses. Les disse for å forstå dagens struktur før refaktoreringen.

## Suksesskriterier

✅ Alle unit tests kjører med in-memory test-implementasjoner (ingen ekte database/filsystem)
✅ Domene-modeller er UI- og database-uavhengige, samlet i Models-namespace
✅ Services bruker constructor injection av interfaces
✅ Ingen static singletons for data eller tjenester
✅ Ingen adapter-hack eller workarounds
✅ All eksisterende funksjonalitet fungerer uendret
✅ Koden kompilerer i .NET 10

## Del 2: Interface-Basert Dependency Injection

**Opprett `IDatabaseService` interface** med minimum disse operasjonene (utvid basert på metodekall i vedlagte filer):

```csharp
public interface IDatabaseService
{
    // Session management
    Task SaveSessionAsync(SessionData data);
    Task<List<SessionData>> LoadSessionsAsync(int userId);
    Task<List<TrainingSession>> GetTrainingSessions(int userId, DateTime? startDate = null);
    Task<TrainingSession?> GetSessionById(int sessionId);
    
    // Complexity progression
    Task SaveComplexityProgressAsync(ComplexityProgress progress);
    Task<ComplexityProgress> LoadComplexityProgressAsync(int userId);
    
    // SmartCoach baseline & goals
    Task SaveBaselineAsync(SmartCoachBaseline baseline);
    Task<SmartCoachBaseline?> GetBaselineAsync(int userId);
    Task SaveGoalsAsync(List<SmartCoachGoal> goals);
    Task<List<SmartCoachGoal>> GetActiveGoalsAsync(int userId);
    
    // Health monitoring
    Task<List<HealthIssue>> GetRecentHealthIssuesAsync(int userId, int days);
    Task SaveHealthIncidentAsync(HealthIncident incident);
    
    // Gamification
    Task SaveUserProgressAsync(UserProgressData progress);
    Task<UserProgressData> GetUserProgressAsync(int userId);
    Task<List<AchievementData>> GetAllAchievementsAsync(int userId);
    
    // User settings
    Task<UserSettings> GetUserSettingsAsync(int userId);
    Task SaveUserSettingsAsync(UserSettings settings);
    
    // FemVoiceScore results
    Task SaveFemVoiceScoreResultAsync(FemVoiceScoreResult result);
    Task<List<FemVoiceScoreResult>> GetRecentScoresAsync(int userId, int count);
}
```

**Implementer `DatabaseService : IDatabaseService`** med eksisterende SQLite-logikk.

**Implementer `TestDatabaseService : IDatabaseService`** med:
- In-memory collections (`Dictionary<int, List<T>>` for user-scoped data)
- Thread-safe operasjoner (bruk `ConcurrentDictionary` hvis nødvendig)
- Ingen file I/O eller database-kall
- Synkron returnering av Task-wrappede resultater for unit tests

## Del 5: Absolutte Arkitekturregler

**Systemet skal IKKE:**

❌ Ha domene-modeller inne i servicefiler eller DatabaseService
❌ Bruke static singletons for data eller tjenester (Instance-properties)
❌ Bruke adapter/wrapper-hacks for å omgå dependency injection
❌ Koble services direkte til konkrete DatabaseService, LocalizationService, eller andre service-klasser
❌ Ha unit tests som krever ekte database, filsystem, eller nettverkstilgang

**Systemet skal:**

✅ Bruke constructor injection konsekvent
✅ Ha klare interface-boundaries mellom lag
✅ Følge SOLID-prinsipper (spesielt Dependency Inversion)
✅ Tillate full testbarhet med in-memory implementasjoner
✅ Separere domene-logikk fra infrastruktur (persistence, UI, I/O)

FemVoice Studio skal etter refaktoreringen ha profesjonell arkitektur klar for langsiktig vedlikehold og utvidelse:

- **Ren domene-logikk** isolert i Models-namespace
- **Full testbarhet** uten eksterne avhengigheter

- **Vedlikeholdbar kodebase** uten teknisk gjeld fra tight coupling
- **Stabil plattform** for videre kliniske features

Dette er fundamentet for videre utvikling av adaptive training algorithms, machine learning integration, og cloud-syncing capabilities.
